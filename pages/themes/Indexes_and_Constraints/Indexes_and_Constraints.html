MySQL Indexes

Introduction

What and Why?

A database index is added to a table to speed up the search operation.
An index is used by the Database Search Engine to locate records very fast.
Indexes can be created for one or more columns.
Indexes are implemented as a type of tables, which keep index field and a pointer to each record into the actual table.
While adding index to a table will speed the SELECT queries, it will slow down a bit the INSERT and UPDATE statement.

Types of indexes

A regular INDEX
Regular INDEX main use case is to index column(s) which will be searched most often
We can add a UNIQUE constraint to regular INDEX to specify that all rows of the index must be unique. That is, the same row may not have identical non-NULL values for all columns in this index as another row
PRIMARY KEY
The PRIMARY KEY uniquely identifies each record in a table.
In contrast to regular INDEX, PRIMARY KEY can not contain NULL val
FULLTEXT
FULLTEXT indexes are created on text-based columns (CHAR, VARCHAR, or TEXT columns) to speed up queries and text-based operations.

Note that in MySQL KEY and INDEX are synonyms.

Regular index

Add Regular Index to existing table

There are two ways we can add an index to existing table:

With Alter Table syntax:

				ALTER TABLE table_name ADD INDEX [index_name] (column_name[(length)]);
index_name is optional
Using CREATE INDEX statement:

				CREATE INDEX index_name ON table_name (column_name[(length)]);
We can index one or combination of two or more columns.
For string columns, indexes can be created that use only the leading part of column values, using col_name(length) syntax

Consider adding Regular Index to columns which will be searched most often

Example

Let's have the artist table, defined as:

				mysql music_db_admin@localhost:music_db> desc artist;
				+---------+-------------+--------+-------+-----------+---------+
				| Field   | Type        | Null   | Key   |   Default | Extra   |
				|---------+-------------+--------+-------+-----------+---------|
				| fname   | varchar(20) | YES    |       |    <null> |         |
				| lname   | varchar(20) | NO     |       |    <null> |         |
				+---------+-------------+--------+-------+-----------+---------+
Let's add a regular index on lname column:

				-- with ALTER TABLE:
				ALTER TABLE artist ADD INDEX(lname(10));

				-- Or with CREATE INDEX:
				CREATE INDEX artist ON artist(lname(10));
The result will be:

				mysql music_db_admin@localhost:music_db> desc artist;
				+---------+-------------+--------+-------+-----------+---------+
				| Field   | Type        | Null   | Key   |   Default | Extra   |
				|---------+-------------+--------+-------+-----------+---------|
				| fname   | varchar(20) | YES    |       |    <null> |         |
				| lname   | varchar(20) | NO     | MUL   |    <null> |         |
				+---------+-------------+--------+-------+-----------+---------+
The MUL key means that multiple occurrences of a value may occur within that column.

Adding indexes when creating tables

You should be aware that adding an index to a large table can take a very long time.
That's why it's best practice to add INDEX when creating the table!

				CREATE TABLE artist (
					fname VARCHAR(20) DEFAULT NULL,
					lname VARCHAR(20) NOT NULL,
					INDEX(lname(10))
				);
Note that if we do not give explicit index_name, MySQl will set one:

				MySQL test@(none):Tmp> SHOW CREATE TABLE artist;
				+--------+--------------------------------------------------------------------+
				| Table  | Create Table                                                       |
				+--------+--------------------------------------------------------------------+
				| artist | CREATE TABLE `artist` (                                            |
				|        |   `fname` varchar(20) DEFAULT NULL,                                |
				|        |   `lname` varchar(20) NOT NULL,                                    |
				|        |   KEY `lname` (`lname`(10))                                        |
				|        | ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci |
				+--------+--------------------------------------------------------------------+

PRIMARY KEY

Why using PRIMARY KEY?

Primary Key ensures that a column (or combination of two or more columns) have a unique identity which helps to find a particular record in a table more easily and quickly
Regular indexes do not impose the values to be unique! Neither constraint the values to be NOT NULL.
PRIMARY KEY is an index with both constraints set: UNIQUE and NOT NULL value for each row.
A table can have only one PRIMARY KEY.
As regular indexes a primary key can consist of single or multiple columns (fields).
PRIMARY KEY is very important when we want to split data into different tables with a relation between them.

Create PRIMARY KEY

Best practice it to add PRIMARY KEY on CREATE TABLE.
Note, that CREATE INDEX cannot be used to create a PRIMARY KEY; only ALTER TABLE can be used.
We can add Primary Key within the column definition, like:

				CREATE TABLE t(
					id INT UNSIGNED PRIMARY KEY,
					c1 SMALLINT,
					c2 SMALLINT
				);
We can also add Primary Key within table definition, which allows us to set multiple columns as Primary Key:

				CREATE TABLE t(
					c1 SMALLINT,
					c2 SMALLINT,
					PRIMARY KEY(c1,c2)
				);
As PRIMARY KEY implies UNIQUE and NOT NULL values, these constraints are present, without the need to set them explicitly.

				MySQL test@(none):Tmp> desc t;
				+-------+----------+------+-----+---------+-------+
				| Field | Type     | Null | Key | Default | Extra |
				+-------+----------+------+-----+---------+-------+
				| c1    | smallint | NO   | PRI | (NULL)  |       |
				| c2    | smallint | NO   | PRI | (NULL)  |       |
				+-------+----------+------+-----+---------+-------+

The AUTO_INCREMENT attribute

The PRIMARY KEY values must be UNIQUE and NOT null, but our table not always should have such data, so we add an id filed to serve the purpose of PRIMARY KEY.
In such cases we use the AUTO_INCREMENT attribute on `id` definition, which will generate a unique identity for new rows:

				CREATE TABLE t(
					id INT PRIMARY KEY AUTO_INCREMENT,
					c1 SMALLINT,
					c2 SMALLINT
				);
A column defined with that attribute will set the value of its contents to that of the column entry in the previously inserted row, plus 1.

				insert into t (c1,c2) VALUES (1,1),(1,1),(1,1);

				select * from t;
				+----+----+----+
				| id | c1 | c2 |
				+----+----+----+
				| 1  | 1  | 1  |
				| 2  | 1  | 1  |
				| 3  | 1  | 1  |
				+----+----+----+

SQL Constraints and Foreign Key

SQL Constraints

Overview

MySQL constraints are used to define rules to allow or restrict what values can be stored in columns.
The purpose of inducing constraints is to enforce the integrity of a database.
MySQL supports next constraints:
CONSTRAINT DESCRIPTION
NOT NULL Specify that a column can not contain any NULL value.
UNIQUE Does not allow to insert a duplicate value in a column. But more than one UNIQUE columns can be used in a table.
PRIMARY KEY A PRIMARY KEY constraint for a table enforces UNIQUE and NOT NULL
FOREIGN KEY FOREIGN KEY is used for cross-referencing related data across tables. FOREIGN KEY constraints keep the related data consistent.
CHECK A CHECK constraint controls the values in the associated column. The CHECK constraint determines whether the value is valid or not from a logical expression.
DEFAULT In a MySQL table, each column must contain a value ( including a NULL). While inserting data into a table, if no value is supplied to a column, then the column gets the value set as DEFAULT.

NOT NULL

Indicates that a column cannot store NULL value

				CREATE TABLE artist (
					fname VARCHAR(20),
					lname VARCHAR(20) NOT NULL
				);
			

				INSERT INTO artist (fname) VALUES ('Robert');

				--ERROR: Field 'lname' doesn't have a default value

DEFAULT

Specifies a default value for a column

				CREATE TABLE artist (
					fname VARCHAR(20),
					lname VARCHAR(20) NOT NULL DEFAULT ''
				);
			

				INSERT INTO artist (fname) VALUES ('Robert');
				-- Query OK, 1 row affected

				SELECT * FROM artist;
				+--------+-------+
				| fname  | lname |
				+--------+-------+
				| Robert |       |
				+--------+-------+

UNIQUE

Ensures that each row for a column must have a unique value

				CREATE TABLE users(
					pin INT NOT NULL UNIQUE,
					name VARCHAR(20)
				);

				INSERT INTO users VALUES (1234561234, 'Jhon');
				-- Query OK, 1 row affected

				INSERT INTO users VALUES (1234561234, 'Adam');
				--ERROR: Duplicate entry '1234561234' for key 'users.pin'
Some vendors, MySQL as well, allows a UNIQUE column to be set with DEFAULT NULL values, as by definition a NULL value is not equal to a NULL value. Anyway, it is not recommended to have UNIQUE and DEFAULT NULL column.

CHECK

Full support of CHECK constraint is introduced in MySQL 8.0.16
The CHECK constraint is used to limit the value range that can be placed in a column
CHECK constraint can be defined in a column definition - it will allow only certain values for this column

			CREATE TABLE user(
				age SMALLINT CHECK(age>0 && age<150)
			);
CHECK constraint can be defined on a table - to limit the values in certain columns based on values in other columns in the row.

			CREATE TABLE t(
				c1 SMALLINT CHECK(c1>0),
				c2 SMALLINT,
				CHECK (c2-c1>0)
			);

FOREIGN KEY constraints

What is a FOREIGN KEY?

A FOREIGN KEY in one table points to a PRIMARY KEY in another table.
FOREIGN KEY relationships involve a parent table that holds the central data values, and a child table with identical values pointing back to its parent.
The FOREIGN KEY clause is specified in the child table. The parent and child tables must use the same storage engine. They must not be TEMPORARY tables.
Reference: FOREIGN KEY Constraints

			-- Define the Parent Table
			CREATE TABLE `customers` (
				`id` int NOT NULL AUTO_INCREMENT,
				`first_name` varchar(45) NOT NULL,
				`last_name` varchar(45) NOT NULL,
				`email` varchar(100) DEFAULT NULL,
				PRIMARY KEY (`id`)
			);
		

			-- Define the Child Table
			CREATE TABLE `orders` (
				`id` int NOT NULL AUTO_INCREMENT,
				`customer_id` int NOT NULL,
				`order_date` datetime NOT NULL,
				PRIMARY KEY (`id`),
				FOREIGN KEY (`customer_id`) REFERENCES `customers` (`id`)
			);

Example on child table insert

Note, that if we try to insert into orders the id of a customer, who does not exists, MySQL will through an error

			INSERT INTO orders (customer_id,order_date)
				VALUES (1, "2020-01-20 21:00:00");

			--ERROR: Cannot add or update a child row: a foreign key constraint fails

customersdb ER Diagram

This implements "one to many" relation!
One customer can have many orders.
If we need "many to many" relation we could make third table: customers_orders which will store foreign keys to customers(id) and orders(id)

FOREIGN KEY Constraint: ON DELETE

ON DELETE specifies what happens to the child table when parent table rows are deleted.
ON DELETE CASCADE causes matching child rows to be deleted when the corresponding parent row is deleted.
In essence, the effect of the delete is cascaded from the parent to the child. This enables you to perform multiple-table deletes by deleting rows only from the parent table and letting InnoDB delete the corresponding rows from the child table.
For MySQL InnoDB tables ON DELETE NO ACTION and ON DELETE RESTRICT are the same as omitting the ON DELETE clause,as the foreign key constraints are checked immediately.
ON DELETE SET NULL causes index columns in matching child rows to be set to NULL when the parent row is deleted. If you use this option, all the indexed child table columns named in the foreign key definition must be defined to permit NULL values

FOREIGN KEY Constraint: ON UPDATE

ON UPDATE enables you to specify what happens to the child table when parent table rows are updated.
If the parent primary key is changed, the child value will also change to reflect that.
If ON UPDATE clause is not present, the default behaviour is to reject any inserts or updates in the child table that result in foreign key values that don’t have any match in the parent table index, and to prevent updates to parent table index values to which child rows point.
The possible action values are the same as for ON DELETE and have similar effects.

FOREIGN KEY Constraint: Guidelines

You must create the parent table index explicitly before defining the foreign key relationship.
InnoDB automatically creates an index on foreign key columns (the referencing columns) in the child table if the CREATE TABLE statement does not include such an index.
The automatically created index will be a non unique index and will include only the foreign key columns.
You should define the index in the child table explicitly if you want it to be a PRIMARY KEY or UNIQUE index, or if it should include other columns in addition to those in the foreign key.
Corresponding columns in the parent and child indexes must have compatible types.
Corresponding character columns must be the same length.
Corresponding integer columns must have the same size and same range (signed or unsigned).
For string columns, you must index the entire column, not just a leading prefix of it.
